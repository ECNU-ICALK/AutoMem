# AutoMem Architecture Search — Round {{ round_id }} / {{ max_rounds }}

You are an expert memory-system architect for LLM agents.

**Goal**: Across {{ max_rounds }} rounds of evolution, find the memory architecture
that maximizes task-accuracy lift over the no-memory baseline on the {{ benchmark_name }} benchmark. Each round

## 1. Search Space (4 layers)

An architecture is a 4-tuple: `extract_types` × `storage_routing` × `retrieval`
× `management`. Each layer below lists the allowed options; nothing outside
these tables is valid.

### 1A. extract_types — which memory representations to extract (pick a non-empty subset)

| Type       | Fires on      | What it captures                                            |
|------------|---------------|-------------------------------------------------------------|
| tip        | any task      | Domain-agnostic cognitive heuristics (how-to-think rules)   |
| trajectory | any task      | Compressed action→observation fact chains (what happened)   |
| workflow   | success only  | Reusable orchestration logic (sequencing + tool scheduling) |
| shortcut   | any task      | Executable macro actions with parameterized placeholders    |
| insight    | failure only  | Root-cause failure analysis + failure knowledge graph       |

`workflow` is auto-gated to successful tasks; `insight` to failures. Select a
non-empty subset of 1 to 5 unique types for each architecture — e.g.
`["tip", "trajectory", "workflow"]` combines heuristics, episodes, and
orchestration logic. Larger subsets capture more knowledge per task but raise
extraction cost and pool noise; let the diagnosis evidence justify each added type.

### 1B. storage_routing — where the selected extract types are persisted

Choose exactly one backend and route EVERY selected type to it (one common
store per architecture; mixed backends are outside this search space):

| Backend   | Cold-start threshold | Mechanism                                                 |
|-----------|----------------------|-----------------------------------------------------------|
| json      | 5 units              | Exact key lookup                                          |
| vector    | 20 units             | FAISS semantic similarity                                 |
| hybrid    | 15 units             | json + vector union (returns exact AND semantic matches)  |
| graph     | 30 units             | NetworkX entity-relation graph (N-hop traversal)          |
| llm_graph | 50 units             | LLM-constructed knowledge graph; highest fidelity & cost  |

Before choosing a backend, check §2A pool_stats. If your per-type unit count is
below a backend's threshold, retrieval on that backend will return empty until
the pool grows.

### 1C. retrieval — how memories are fetched at query time (pick one)

| Strategy     | Requires graph storage | Mechanism                                            |
|--------------|------------------------|------------------------------------------------------|
| hybrid       | No                     | Semantic embedding + lexical union                   |
| contrastive  | No                     | Per-unit scoring via positive vs negative cohorts    |
| cbr_rerank   | No                     | Case-based candidate pool + LLM rerank               |
| graph        | YES                    | Entity-relation graph traversal (multi-hop)          |
| hyde         | No                     | LLM-hypothesized answer passage + averaged-embedding dense retrieval (Gao et al. 2022). Strong on long-tail / complex-task reasoning where the literal query is short. Adds 1 LLM call per query. |
| mmr          | No                     | Dense recall + Maximal-Marginal-Relevance diversity re-rank (Carbonell & Goldstein 1998). 0 extra LLM calls. Best when the pool has many near-duplicates. |

### 1D. management — memory-lifecycle preset (pick one)

Every preset is automatically wrapped with the 8 mandatory ops (see §1E). The
preset below selects the *additional* experimental ops on top.

| Preset          | Additional ops                                                                                          | Storage requirement |
|-----------------|---------------------------------------------------------------------------------------------------------|---------------------|
| lightweight     | time_decay + score_based_prune                                                                          | any                  |
| json_full       | cluster_merge + reflection_correction + cross_type_dedup + trajectory_to_workflow + reindex_relations   | any                  |
| tool_manager    | shortcut_promotion (post-task) + shortcut_validation (on-insert); best when the selected extract type is `shortcut` | any              |
| graph_consolidate | graph content consolidation + cross-task generalization + success-aware edge adaptation; excludes shortcut ops | graph or llm_graph storage + graph retrieval |

(Op semantics → §1F glossary.)

### 1E. Mandatory layer (always-on, not selectable)

Every architecture automatically includes these 8 ops regardless of preset:
LLM Judge (post-retrieval relevance filter), signature_dedup, conflict_detection,
quality_curation, utility_audit, access_stats_update, boost_on_success,
penalize_on_failure. Do NOT include these in your output — they are guaranteed.
A "lightweight" preset still gets all 8.

### 1F. Op glossary (reference; you do NOT select these directly)

| Op                       | What it does                                                                                       |
|--------------------------|----------------------------------------------------------------------------------------------------|
| `time_decay`             | Down-weights memory scores by age                                                                  |
| `score_based_prune`      | Caps the pool size by keeping only top-K by score                                                  |
| `cluster_merge`          | Hierarchically clusters similar memories; replaces each cluster with its centroid                  |
| `reflection_correction`  | Detects contradicting pairs; LLM picks which one wins                                             |
| `cross_type_dedup`       | Prevents the same fact stored as multiple types (e.g. both tip AND trajectory)                     |
| `trajectory_to_workflow` | Promotes recurring successful trajectories into reusable workflow units                            |
| `reindex_relations`      | Rebuilds graph indices after bulk inserts / deletions                                              |
| `cross_task_generalize`  | Detects patterns recurring across distinct tasks, lifts them into abstract reusable memory         |
| `shortcut_promotion`     | Boosts retrieval priority of shortcuts that were used in successful tasks                          |
| `shortcut_validation`    | Validates a shortcut still produces correct output before persisting on insert                     |

## 4. Output format

Return strict JSON — an array with exactly {{ num_candidates }} objects (or
{{ num_candidates_for_llm }} starting from `candidate_id: 1` when a champion is
auto-injected this round). Each object follows this schema:

```json
[
  {
    "candidate_id": 0,
    "hypothesis": "One sentence: what specific weakness this candidate addresses.",
    "architecture": {
      "extract_types": ["tip", "shortcut"],
      "storage_routing": {"tip": "json", "shortcut": "json"},
      "retrieval": "contrastive",
      "management": "tool_manager"
    },
    "diversity_role": "exploit | explore_retrieval | explore_extract | explore_storage | validate_open_question | exploit_a | exploit_b | exploit_c",
    "rationale": "2-3 sentences connecting pool_stats, attribution diagnosis, and this design choice. For exploit roles, list which layers were inherited from the champion and which were mutated."
  }
]
```

**Validation rules**:
- `architecture` has exactly 4 keys: `extract_types`, `storage_routing`,
  `retrieval`, `management`. No other keys (no `injection` — injection is fixed
  at the runtime layer to raw_topk and is not a search dimension).
- `extract_types`: a non-empty array of 1 to 5 UNIQUE types from §1A
  (e.g. `["tip"]` or `["tip", "trajectory", "workflow"]`).
- `storage_routing`: exactly one key per selected extract type, and every key
  mapped to the SAME backend from §1B (one common store; mixed backends are
  invalid).
- `retrieval`: exactly one of the 6 strategies in §1C. If you pick `graph`,
  the common `storage_routing` value must be `graph` or `llm_graph`.
- `management`: exactly one of the 4 presets in §1D. If you pick
  `graph_consolidate`, retrieval must be `graph` and the common
  `storage_routing` value must be `graph` or `llm_graph`.
- For any `exploit*` role: at least 2 of the 4 layers must be unchanged from
  the current champion.
- Cold-start: if `pool_stats` shows a per-type count below a backend's threshold,
  prefer a lower-threshold backend for that type.
- Output strict JSON — no comments, no trailing commas.
