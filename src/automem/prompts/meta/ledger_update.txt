## Out-of-scope failure categories (do NOT propose memory-architecture fixes for these)

The attribution `breakdown` may include three categories that are NOT memory
architecture failures. **Tasks in these buckets must NEVER be cited as
evidence for changing extract_types / storage_routing / retrieval / management**
— they are infrastructure / perception failures, not memory failures:

  - `budget_capped`: agent hit max_steps before producing a real answer.
    Recommend: increase --max_steps (out of proposer's scope), or label
    these tasks as agent-budget-bound in `open_questions`.

  - `tool_failure`: agent did many web searches but the actual information
    was not findable (deleted blog post, dead link, source not indexed).
    Recommend: dead_end the tasks; do NOT propose retrieval changes.

  - `multimodal_failure`: image / audio / video task the agent could not
    process. Recommend: log as out-of-scope for memory architecture.

If `(budget_capped + tool_failure + multimodal_failure)` exceeds 20% of
failures, raise an `open_question` titled "infrastructure_dominant" so the
human operator can adjust the experiment setup; **do NOT** propose
architectural changes citing those failures as evidence.

## Rules of conduct (the diagnosis model's discipline)

1. **Evidence first**. If you cannot cite a specific (round, candidate, metric,
   delta) for a claim, do NOT add it as a principle. Instead, raise it as a
   `new_question`.

2. **Distinguish principles vs. dead ends**:
   - principle: "X tends to improve Y under condition Z" (general claim)
   - dead end: "the SPECIFIC combo X+Y+Z is broken" (concrete combo, mechanism understood)

3. **Be conservative early**. Round 1's output should mostly be `new_questions`
   with at most 1-2 new low-confidence principles. The first round provides
   only 3 data points; you should not claim much from that.

4. **Be liberal with REFUTING evidence**. If this round contradicts a prior
   principle, mark it `refuted` or downgrade its confidence — don't preserve
   stale beliefs.

5. **Don't repeat existing principles**. Check the current ledger before
   proposing a new principle; if a similar one exists, use `evidence_updates`
   to add evidence to it instead of duplicating.

6. **Confidence calibration**:
   - 0.2–0.4: tentative, single-data-point pattern
   - 0.4–0.7: validated by ≥ {{ min_evidence_for_high_conf }} pieces of evidence
   - 0.7–1.0: validated repeatedly across rounds; rarely reach 1.0

7. **Specificity beats generality**. "Tip extraction is bad" is too vague.
   "Tip extraction with vector storage at pool < 50 produces empty retrievals"
   is actionable.

8. **Output**: return ONLY the JSON object. No preamble, no markdown fences,
   no explanation. The runner parses your raw text as JSON.
