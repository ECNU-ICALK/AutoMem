## Synthesis rules

You **MUST** follow these rules when resolving conflicts:

**Rule 1 — Cross-source conflict resolution**:
When `rule_diagnosis` and `layer_diagnosis` disagree about the primary
bottleneck, **TRUST `layer_diagnosis`** (it has seen evidence + historical
context, while rule is a count-based template). Reflect this in
`cross_source_agreement`.

**Rule 2 — memory_bypass detection**:
Set `memory_bypass = TRUE` when BOTH:
- `memory_compliance.avg_followed_score < 0.5`
- AND some dim in `memory_compliance.violation_distribution` is ≥ 0.3

This means memory was retrieved correctly but agent ignored its instructions.
Memory mutation cannot fix this — the agent is the bottleneck.

**Rule 3 — out_of_scope_for_memory detection**:
Set `out_of_scope_for_memory = TRUE` when EITHER:
- The dominant failure class in `breakdown` is one of:
  `reasoning_error` / `near_miss` / `numeric_precision` / `time_range_mismatch` /
  `judge_format_mismatch` / `budget_capped` / `tool_failure` / `multimodal_failure`
- OR `memory_bypass = TRUE`

This means memory architecture changes are unlikely to fix the dominant
failures. Recommend "propose diverse explore_* candidates" rather than
chasing this specific failure mode via retrieval/storage/management.

**Rule 4 — confidence assignment**:
- `high`: rule and LLM agree about the bottleneck AND historical trend is consistent
- `medium`: rule and LLM disagree (you trusted LLM per Rule 1) OR no historical data
- `low`: signals contradict in non-obvious ways OR sample size too small to trust

**Rule 5 — evidence_task_ids**:
Extract 2-3 specific `task_id`s that the LLM 4-layer observation cited.
These will be the primary evidence the proposer reasons over.

**Rule 6 — recommended_action**:
- If `memory_bypass = TRUE`: write "Memory IS effective but agent ignores instructions on D{N}.{dim_name} — do NOT mutate retrieval/storage/management; propose only explore_* candidates that exercise under-tested architecture dimensions."
- If `out_of_scope_for_memory = TRUE`: write "Primary failure mode ({class_name}) is not memory-addressable — propose diverse architectures (explore_extract / explore_storage / explore_retrieval) rather than tuning the current candidate's bottleneck."
- Otherwise: cite a **specific architecture-space delta** from
  `layer_diagnosis.priority_action`. Phrase as `<layer>: <current> → <suggested>`,
  e.g. "retrieval: contrastive → hybrid" or "storage_routing.workflow: vector → graph".
  When historical trend shows the same bottleneck for 2+ rounds, emphasize urgency.
